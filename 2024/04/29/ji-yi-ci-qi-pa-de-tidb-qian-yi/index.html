记一次奇葩的TIDB迁移

TIDB 迁移 TIDB-lightning TICDC
发布日期:   2024-04-29
更新日期:   2024-04-30
文章字数:   1.3k
阅读时长:   5 分
阅读次数:  

TIDB 官网文档

https://docs.pingcap.com/zh/tidb/v6.5/ecosystem-tool-user-guide

需求

从旧的 TIDB（生产使用）迁移到新的 TIDB（后面生产使用），并且其中几张表要通过 TICDC 同步到 ES。（ 9 亿数据量，同步完成时长 1 天）

迁移方案

  1. 先使用 BR 工具 将数据迁移到新的 TIDB 集群 （BR工具官网文档地址：https://docs-archive.pingcap.com/zh/tidb/v4.0/backup-and-restore-tool）
  2. 由于 BR 迁移是物理迁移，是触发不了 TICDC 任务的，所以我们单独对这 4 张表做一个处理，使用 Dumpling && Lightning 工具进行备份恢复，先创建好 TICDC（新的临时库）任务，然后使用 Lightning 工具进行恢复，CDC 的 sink 为 kafka，filter_rules 为临时库，通过 logstash 同步到 ES。（Dumpling && Lightning 官网文档地址：https://docs.pingcap.com/zh/tidb/v6.5/dumpling-overview https://docs.pingcap.com/zh/tidb/v6.5/tidb-lightning-overview）
  3. 等待 2 中数据同步到 ES 完成后，删除 2 中的 临时 TICDC 任务，创建新的 TICDC 任务，filter_rules 为正式库，此时在旧的 TIDB 上 创建 TICDC 任务，sink 为 新的 TIDB，start_ts 为 BR 时提示的 BackupTS，这时候就可以完成 TIDB 集群的迁移，并且 数据全部同步到 ES。

解决详情

全量迁移

先拉长 GC 时长，保证迁移期间 BackupTS 可用

SET GLOBAL tidb_gc_life_time = '720h';

然后在上游使用 BACKUP 工具进行全量迁移
由于我们是在生产上去备份，为了防止影响生产，我们做了个限速,RATE_LIMIT = 40 MB/SECOND，将影响控制在最小

BACKUP DATABASE * TO 's3://devops/tidb-prod/?region=us-west-1&access-key=xxx&secret-access-key=xxx' RATE_LIMIT = 40 MB/SECOND;

执行完记录 BackupTS 的值

下游集群中执行 RESTORE 语句备份数据(时间大概为BACKUP的2.5倍，建议放到后台去执行，避免中断)

RESTORE DATABASE * FROM 's3://devops/tidb-prod/?region=us-west-1&access-key=xxx&secret-access-key=xxx'

创建 同步到 ES 的 logstash（临时库）

logstash 配置涉及处理字段细节，略过，有想交流的可以私信我
重点配置（3个节点）
pipeline.worker = 1
input 配置
consumer_threads => 20
因为我的partition有 60 ，3个节点
所以每个节点 20 个线程，并且 worker 为 1，防止 INSERT 跟 UPDATE 之间顺序错乱
如果有更好的处理方法，欢迎私聊我告知我解决方案，谢谢!

创建 CDC 任务 （临时库）

curl -X POST -H "Content-type: application/json" http://127.0.0.1:8300/api/v1/changefeeds -d '{"changefeed_id":"record-sync-qy-all","sink_uri":"kafka://b-1.xxx:9092,b-3.xxx:9092,b-2.xxx:9092/record-sync-qy?protocol=canal-json&kafka-version=2.8.1&partition-num=60&max-message-bytes=67108864&replication-factor=1","filter_rules":["test_record.tbsendrcd"],"sink_config":{"dispatchers":[{"matcher":["test_record.tbsendrcd"], "dispatcher":"rowid"}],"protocol":"canal-json"}}'

dumpling 逻辑备份

### 如果有多张表，再加 -T 'database.table' 就行，由于我这里的数据比较敏感，就不放上来了
tiup dumpling -uroot -p'xxx'  -P4000 -h'tidb-ip' --filetype sql -t 1 -o /data1/test  -r 200000 -F256MiB -T 'record.tbsendrcd'

恢复数据

cd /data1/test
rename record. test_record. *.sql
mv record-schema-create.sql test_record-schema-create.sql
# 再修改 创建库的 SQL 文件，将库名修改为 test_record

# 恢复数据
nohup tiup tidb-lightning -config tidb-lightning.toml > nohup.out &

tidb-lightning.toml

[lightning]
# 日志
level = "info"
file = "tidb-lightning.log"
max-size = 512 # MB
max-days = 28
max-backups = 14

# 启动之前检查集群是否满足最低需求。
check-requirements = false

[mydumper]
data-source-dir = "/data1/test" 
filter = ['*.*', '!mysql.*', '!sys.*', '!INFORMATION_SCHEMA.*', '!PERFORMANCE_SCHEMA.*', '!METRICS_SCHEMA.*', '!INSPECTION_SCHEMA.*']


[tikv-importer]
# 导入模式配置，设为 tidb 即使用逻辑导入模式
backend = "tidb"

[tidb]
# 目标集群的信息。tidb-server 的地址，填一个即可。
host = "tidb-ip"
port = 4000
user = "root"
# 设置连接 TiDB 的密码，可为明文或 Base64 编码。
password = "xxx"
# tidb-lightning 引用了 TiDB 库，并生成产生一些日志。
# 设置 TiDB 库的日志等级。
log-level = "error"

[checkpoint]
# 启用断点续传。
# 导入时，TiDB Lightning 会记录当前进度。
# 若 TiDB Lightning 或其他组件异常退出，在重启时可以避免重复再导入已完成的数据。
enable = true

这时候就可以看到 ES 有数据写入了

创建 CDC 任务 （正式库）

历史数据同步完后，先删除之前到 TICDC（临时库）

curl -X DELETE http://127.0.0.1:8300/api/v1/changefeeds/record-sync-qy-all

再创建新的 TICDC（正式库），同时也要再创建正式库的 logstash
目前暂时不会有数据进来，因为我们现在新的 TIDB 是没有任何数据 CRUD 的，等到我们创建 TIDB（旧）到 TIDB（新）的 CDC 任务后，就会开始同步数据了

curl -X POST -H "Content-type: application/json" http://127.0.0.1:8300/api/v1/changefeeds -d '{"changefeed_id":"record-sync-qy","sink_uri":"kafka://b-1.xxx:9092,b-3.xxx:9092,b-2.xxx:9092/record-sync-qy?protocol=canal-json&kafka-version=2.8.1&partition-num=60&max-message-bytes=67108864&replication-factor=1","filter_rules":["record.tbsendrcd"],"sink_config":{"dispatchers":[{"matcher":["record.tbsendrcd"], "dispatcher":"rowid"}],"protocol":"canal-json"}}'

增量实时同步数据

# 在旧的的 tidb 操作
curl -X POST http://127.0.0.1:8300/api/v1/changefeeds -H 'Content-Type: application/json' -d '{
"changefeed_id": "upstream-to-downstream",
"sink_uri": "mysql://root:xxx@tidb-ip:4000",
"start_ts": 449317637862195222, # BR的时候记录的 BackupTS
"config": {
"enable-old-value": true,
"force-replicate": true
}
}'
# 等数据同步差不多了，恢复 GC 的时间
SET GLOBAL tidb_gc_life_time = '24h';

到这里就完成迁移了，有任何好的建议方案可以私聊我，谢谢大家
